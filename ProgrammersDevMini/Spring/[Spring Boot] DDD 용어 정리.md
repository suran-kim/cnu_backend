# Domain Driven Design(DDD) 용어 정리

도메인 주도 설계의 용어를 학습한다.
<br/>
<br/>


## 🌿 Domain
사용자가 어플리케이션을 사용하는 대상 영역
주로 비즈니스 자체가 `Domain`이 된다.

ex) 
```
주문관리 어플리케이션의 도메인? 
주문

회계관리 어플리케이션의 도메인?
회계
```

도메인(비즈니스)자체가 복잡하기 때문에 이런 복잡성 해소를 위해 `모델`을 만드는데, 
이런 모델들을 표현하고 설계하는 방법으로 다이어그램을 사용한다.
여러 모델 중 `Entity모델`, `Value Object 모델`을 학습해볼 것이다.

DDD의 용어를 통해 주문관리 어플리케이션을 제작해볼 예정이다.

지금 모델을 학습하는 목적은 JAVA 기반의 어플리케이션을 제작하기 위함이므로 
모든 모델은 클래스로 작성하고 객체로 만들어진다. 종류가 있는 객체라고 생각하면 이해가 쉬울 것.

<br/>
<br/>


## 🌿 Entity

Entity모델에서의 객체는 `Entity`이고 
`Entity`는 다른 Entity와 구별할 수 있는 `식별자`를 가진다.
Entity는 **객체의 상태가 계속 변한다**. 
Entity의 `식별자`는 고유하기 때문에 `도메인`에서 개별성이 있는 개념이라고 생각하면 된다.
```
EX) 
주문관리 도메인에서는 
주문이라는 Entity 존재. 각각의 주문은 독립적이고, 유니크하고, 개별성있다. 
고객이라는 Entity 존재. 고객은 고객이라는 Entity를 식별할 수 있는 `고객 ID`를 가지고 있다,
```
주문관리 도메인에서 주문의 상태는 계속 변할 수 있다. 
상품이 배송 중 -> 배송완료가 되면 주문은 complete상태로 변하는 것이다.
고객의 패스워드, 전화번호 등은 변할 수 있는 속성이며 고객정보는 개별적이고 지속적으로 변화된다.

<br/>
<br/>


## 🌿 Value Object
Value Object는 값 객체라고 불린다. 
Value Objec는 값 속성이 개별적으로 변하지 않고 값 자체로 고유한 객체를 뜻한다. 
엔티티가 `id`로 고유성을 가진다면 Value는 `값의 조합`으로 고유성을 가진다. 
그리고 **한 번 만들어지면 내용이 변하지 않는다. **

ex) 100원짜리 동전은 객체 자체가 고유하다. 
100원 동전을 5개 가지고 있으면 100원 동전 객체 5개가 각각 고유한 것처럼 Value Object도 고유하다. 
한 번 100원은 영원히 100원이다.

대체로 `entity`들이 속성으로 `Value Object`를 가지고 있다.

> _주문 시스템에서 주문자는 변하지 않는다. 
`주문 엔티티`가 사라진다면 주문 엔티티와 함께 주문자도 사라질 것이다.
그러나 주문자의 이름이 변하는 것은 아니다.
배송지에 변경이 일어났을 때에도 배송지 안의 주소가 변하는 것이 아니라 
주문의 배송지로서 다른 Value Object가 들어가는 것이다. 
그런데 배송지 자체가 `주소`라면 주소는 그 자체로 고유한 것이 된다. 
`주문물품 엔티티`도 마찬가지다. 주문물품의 목록이 있다면 목록은 바뀔 수 있지만 
하나의 `Value Object`는 한 번 만들어지면 변하지 않으므로 
주문물품 안의 속성을 변경할 수는 없는 것이다._
